A Benchmark-to-Audit Framework for Source-Code Watermarking in Code Generation Models

Problem

Source-code watermarking is an evidence problem.

When generated code is copied, edited, translated, or deployed, the original prompt session is no longer enough to support a provenance claim.

01

Code must execute

For source code, a watermark result is weak if the benchmark row is not tied to executable behavior and a stable denominator.

02

Access changes meaning

White-box recovery, black-box audit, owner attribution, and security triage cannot share one accuracy score.

03

Abstention is evidence

If a row cannot support a strong claim, the framework keeps it as null, support-only, or review evidence instead of forcing a label.

Why this matters Executable rows protect the denominator.

In the viva, this lets me say exactly which code-generation rows were counted before interpreting any watermark signal.

Oral opening

My project is not trying to prove one universal watermark detector. The core idea is evidence admission: before I say what a result means, I fix the denominator, controls, artifact, access model, and claim boundary.

Evidence contract

The central contribution is a rule for what a watermark result may claim.

What is counted?

A result only means something after the denominator is fixed.

For example, SemCodebook reports 30,330 recoveries over 31,200 admitted positive rows. Misses remain inside the denominator; they are not hidden by the aggregate.

Limits and examiner-safe claims

The strongest part of the work is that it says what it does not prove.

No universal watermark claim

SemCodebook supports admitted white-box cells; it is not a universal natural-generation theorem.

No provider accusation

CodeDye reports conservative black-box audit evidence; it is not contamination prevalence or wrongdoing evidence.

No general authorship proof

ProbeTrace is scoped active-owner verification under a fixed registry and split.

No safety certificate

SealAudit is selective triage with abstention; it is not automatic harmlessness certification.

Q&A landing page

Stop here after the walkthrough.

Short answers are enough. For most questions, answer with one claim, one denominator, and one boundary.

What is the novelty?

WatermarkScope makes code watermarking evidence auditable by tying each result to denominator, controls, artifact, access model, and claim boundary.

Which part is the main method?

SemCodebook is the main method contribution: structured white-box provenance recovery using typed program carriers and keyed scheduling.

Why not one accuracy score?

Benchmark completion, provenance recovery, black-box audit, attribution, and triage answer different questions under different access assumptions.

What would you improve next?

I would add new admitted evidence surfaces rather than changing old denominators: more model cells, stronger black-box calibration, and broader owner registries.
